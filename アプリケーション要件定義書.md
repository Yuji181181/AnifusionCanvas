# **アプリケーション開発用詳細要件定義書**

## **開発の背景と目的**

アニメーション制作産業は世界的な需要の拡大を背景に成長を続けているが、その制作現場は歴史的に極めて労働集約的なプロセスに依存している。特に、原画と原画の間を補間して滑らかな動きを作り出す「中割り」の工程は、膨大な作画枚数を必要とし、制作スケジュールの遅延やクリエイターの過重労働を引き起こす最大のボトルネックとなっている。

これまでの映像制作業界では、実写映像向けのフレーム補間技術を用いた自動化が試みられてきたが、アニメーションのフレームは時間的に疎（Sparse）であり、非線形な大きな動きや隠蔽（オクルージョン）を伴うため、従来技術では不自然な破綻が必然的に生じていた。

本プロジェクトは、これらの課題を解決する「アニメ制作DX」を実現するためのプロトタイプ（デモ）開発である。最新の生成AI技術とモダンなWebフロントエンド技術を統合した「Human-in-the-Loop（人間とAIの協調）」のワークフローを構築し、現場のクリエイターが実務で利用可能なレベルの支援ツールを提案する。

また、本デモアプリケーションの開発は、エンターテインメントの基盤と高度な技術力を併せ持つ企業に対する、開発者の強い熱意とフルスタックなAIエンジニアリング能力をアピールするためのポートフォリオとして機能することを最大の目的としている。

## **システムアーキテクチャおよび技術スタック**

本アプリケーションは、運用コストの最適化とデモとしての軽快な動作を両立するため、フロントエンドとバックエンドの責務を完全に分離したアーキテクチャを採用する。以下の技術スタックを厳守して実装すること。

| コンポーネント / 領域 | 指定技術および制約 |
| :---- | :---- |
| **フロントエンド基盤** | **Next.js (App Router)** 実行環境およびパッケージマネージャーには **bun** を使用すること。 |
| **データフェッチ・通信** | Next.jsのバックエンド機能（Route HandlersやServer Actionsなど）は基本使用せず、純粋なフロントエンドアプリケーションとして構築する。バックエンドのFastAPIサーバーへ直接通信を行い、APIのポーリングやデータ取得には **useSWR** を使用する。 |
| **スタイリング** | **TailwindCSS** を使用してモダンでレスポンシブなUIを構築する。 |
| **バリデーション** | **Zod** を使用し、フロントエンドの入力検証およびAPIのリクエスト/レスポンスの型安全性を担保する。 |
| **コード品質管理** | リンターおよびフォーマッタとして **Biome** を採用する。 |
| **画像編集・UI** | react-konva 等を用いてキャンバスを実装し、状態管理には Zustand などを活用する。 |
| **バックエンド・API基盤** | **FastAPI** を使用。 Next.js側でバックエンド処理を持たないため、データベースとの接続、非同期ジョブの管理、および外部ストレージ連携などのビジネスロジックはすべてFastAPI側に集約する。 ※APIエンドポイントの具体的な設計は実装担当AIエージェントに委ねる。 |
| **Pythonパッケージ管理** | **uv** を使用する。 依存ライブラリの追加には pip や uv pip install ではなく、必ず **uv add** コマンドを使用し、モダンなプロジェクト管理を行うこと。 |
| **オブジェクトストレージ** | **Cloudflare R2**（アップロードにはFastAPI経由、または署名付きURLを用いたダイレクトアップロード等を採用）。 |
| **データベース** | **TiDB Serverless** ※テーブルのスキーマ構造や、Python側でのORM選定などの具体的な実装内容は実装担当AIエージェントに委ねる。 |
| **GPU推論基盤** | **Modal.com** 等のクラウドGPU、またはローカル環境（FastAPIをホストし、バックグラウンドタスクとしてAI推論を実行）。 |

## **インタラクティブな編集ワークフロー（UI要件）**

本アプリケーションのコアとなるユーザー体験は、AIにすべてを任せるのではなく、クリエイターが介在してクオリティを担保する以下の3ステップのフローで構成される。フロントエンドはこのフローを直感的に操作できるUIを提供すること。

1. **AIで中割りを生成**  
   ユーザーが2枚の原画（キーフレーム）をアップロードし、生成したい枚数を指定すると、バックエンドの動画生成AIがその間のフレームを自動補間し、タイムラインに並べる。  
2. **AIで、その生成した画像に対して自然言語で指示をして修正**  
   生成された中割り画像の中で破綻している部分（例：手の形がおかしい等）があった場合、ユーザーはエディタのキャンバス上でその部分をブラシでマスク（黒塗り）する。その後、「握りこぶし」などの自然言語によるプロンプトを入力して再生成を指示することで、AIがそのマスク部分だけを修正（Inpainting）する。  
3. **それでも修正できなかった部分を、人間の手で直接編集**  
   AIの自然言語修正でもクリエイターの意図に完璧に合致しない1ピクセル単位のズレや色の修正については、キャンバスエディタ上の描画ツールを用いて、人間の手で直接ピクセルを編集・加筆して最終的な完成とする。

## **AIモデルの選定と推論要件**

デモの品質と実装の現実性を両立するため、FastAPIバックエンドで稼働するAI推論パイプラインは以下の戦略に基づいて設計する。

* **中割り生成モデル:** アニメーションのフレーム補間に特化したOSSモデルである **ToonCrafter** を採用する。  
* **部分修正（Inpainting）モデル:** ステップ2の自然言語による修正機能には、**Stable Diffusion v1.5系 または SDXL の Inpainting特化モデル**（アニメ画風に強いモデル、例：AnyLoRA等）を利用する。  
* **キャラクター制御（LoRA適用）:** 別のタスクとして事前学習済みのキャラクターLoRAモデルが用意される想定である。推論時にこのLoRAの重みをベースモデルに適用し、キャラクターのアイデンティティを保った画像を生成できるように実装すること。  
* **IP-Adapter（オプション）:** ユーザーが設定画（リファレンス画像）をアップロードし、その画像特徴量をプロンプトとして直接モデルに注入する機構の導入も推奨する。

## **バックエンド推論処理と非同期ジョブ制御の要件**

長時間を要するAI推論ワークロードを安定して処理するため、FastAPIとNext.jsの間で適切な非同期通信パターンを構築する。

* **ジョブキューとポーリング:**  
  Next.jsからFastAPIへ推論をリクエストした際、FastAPIは即座にジョブIDのみを返却し、裏側（バックグラウンド）で重い推論タスクを実行する。フロントエンドは **useSWR** を用いてFastAPIのエンドポイントを定期的にポーリングし、ジョブの完了ステータスと生成された画像のURLを取得する構成とすること。  
* **GPUメモリの最適化:**  
  ToonCrafter等のモデルはVRAM消費が激しいため、FP16（半精度浮動小数点）でのロードや、不要なモデルのCPUへのオフロードなど、VRAM使用量を抑える工夫をPythonコード内に施すこと。  
* **動画のエンコーディング (FFmpeg):**  
  生成された複数枚のフレームをプレビュー用に一つのMP4動画として書き出す処理を実装する。本アプリケーションはポートフォリオ用のデモであるため、厳密な24fpsや30fpsにこだわる必要はない。ToonCrafterの出力仕様や実装の容易さを最優先とし、最も実装が簡単でエラーが起きにくいフレームレートを実装担当AIが選択してFFmpegのエンコード設定を行うこと。

実装の細部（DBスキーマやAPI仕様など）は本要件定義書の意図を汲み取り、実装を担当するAIエージェントが最も効率的で堅牢な設計を自ら考え、コードとして具現化すること。
